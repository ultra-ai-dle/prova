# /commit — 변경 사항 분석 + 커밋 메시지 생성 + 커밋

## 목적
현재 uncommitted 변경 사항을 분석하여 적절한 커밋 메시지를 생성하고 커밋한다.

## 입력
- `$ARGUMENTS` — (선택) 커밋에 포함할 추가 컨텍스트 또는 의도 설명

## 실행 절차

1. **변경 사항 수집** (병렬 실행):
   - `git status`로 staged / unstaged / untracked 파일 목록 확인
   - `git diff`로 unstaged 변경 내용 확인
   - `git diff --cached`로 staged 변경 내용 확인
   - `git log --oneline -10`으로 기존 커밋 메시지 스타일 파악

2. **변경 없음 처리**:
   - staged + unstaged + untracked 모두 없으면 "커밋할 변경 사항이 없습니다" 출력 후 종료

3. **변경 분석**:
   - 변경된 파일들을 기능/디렉토리 단위로 그룹핑한다.
   - 변경의 성격을 판단한다:
     - `feat` — 새 기능 추가
     - `fix` — 버그 수정
     - `refactor` — 동작 변경 없는 코드 개선
     - `chore` — 설정, 의존성, 빌드 등
     - `docs` — 문서 변경
     - `test` — 테스트 추가/수정
   - `$ARGUMENTS`가 있으면 의도 파악에 참고한다.

4. **스테이징 + 커밋** (확인 없이 바로 실행):
   - 기존 커밋 히스토리의 스타일(언어, 형식)을 따른다.
   - 커밋 메시지는 제목 + 본문 형식:
     ```
     {type}: {제목 — 한 줄 요약}

     {본문 — 무엇을 왜 변경했는지}
     ```
   - untracked/unstaged 파일이 있으면 관련 파일을 `git add`한다.
   - 민감 파일(`.env`, `credentials`, `secret` 등)은 제외하고 경고한다.
   - 생성한 메시지로 바로 `git commit`을 실행한다.

6. **결과 보고**:
   ```
   ✅ 커밋 완료
   {커밋 해시} {커밋 메시지 제목}
   {변경 파일 수} files changed, {추가}(+), {삭제}(-)
   ```

7. **`/review` 제안**:
   - 커밋된 파일 목록에서 feature 경로(`src/` 하위)가 포함되어 있으면 다음을 출력한다:
     ```
     💡 변경된 feature에 /review 를 실행하시겠습니까?
     ```
   - 설정/문서 파일만 변경된 경우(`.claude/`, `docs/`, `package.json` 등)에는 제안하지 않는다.

## 주의사항
- 사용자 확인 없이 바로 커밋한다 — 묻지 않는다.
- `.env`, 인증 정보 등 민감 파일은 자동 스테이징하지 않는다.
- `--amend`는 하지 않는다 — 항상 새 커밋을 생성한다.
