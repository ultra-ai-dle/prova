# /docs-audit — Markdown 문서 전수 정합성 점검·수정

## 입력

- `$ARGUMENTS` — (선택) 특정 `.md` 파일 경로 또는 디렉토리. 미지정 시 저장소 전체(`mocks/` 제외) 스캔.

---

## 배경·목표

- 이 저장소의 **Markdown 문서가 코드·실제 동작과 어긋난 상태**이며, 그 문서를 신뢰해 코딩·문서 작성이 이어지면서 **오류가 전파**되고 있다.
- **목표**: `mocks/` **이하를 제외한** 모든 Markdown을 **현재 코드베이스 기준으로** 검토하고, **불일치·과장·구식 설명을 수정**한다. 본문뿐 아니라 **`.md` 안의 다이어그램(Mermaid 등)** 도 동일하게 **코드·실제 동작과 맞는지** 검증한다(다이어그램은 오해를 크게 키우므로 **특히 엄하게** 본다).
- **원칙**: 기존 Markdown의 주장은 **검증 전까지 신뢰하지 않는다.** 반드시 **소스 코드·설정·타입 정의**와 대조한다.

---

## `/api/explain` 처리 원칙 (삭제 예정)

- **`POST /api/explain`(및 `app/api/explain/`)은 삭제 예정**이다. 저장소에 **라우트 파일이 남아 있어도**, 문서·점검 관점에서는 **제공되지 않는 기능·없는 API**로 취급한다.
- 문서에서 할 일:
  - **제품 기능·아키텍처·다이어그램·클라이언트 흐름**에 explain을 넣지 않거나, 이미 있으면 **삭제·축약**한다.
  - "향후 연동""라우트만 존재" 등 **코드 존재를 전제로 한 설명**도 지양하고, **분석은 `/api/analyze` 중심**만 남긴다.
  - 예외적으로 **히스토리/내부 메모**에 한 줄 남길 때만 "삭제 예정이었던 엔드포인트" 수준으로 적을 수 있다.
- **이번 작업에서 코드를 삭제할 의무는 없다**(문서 정합성 작업이 우선). 다만 문서는 위 정책에 맞춘다.

---

## 범위

- **포함**: 저장소 전체의 `**/*.md` (예: `docs/`, `app/**/CLAUDE.md`, `.cursor/rules/`, `prompts/` 등).
- **제외**: `mocks/` 디렉터리 **내부**의 모든 파일(해당 경로는 조사·수정 대상에서 제외).

---

## 작업 순서 (권장)

### 1. 인벤토리

- `mocks/` 밖의 모든 `.md` 파일 목록을 만든다(경로·대략적 역할 한 줄씩).

### 2. 코드 "진실의 원천" 우선순위

문서 주장과 대조할 때 아래를 우선한다.

- `src/types/prova.ts` — 파이프라인 타입·스키마
- `src/features/execution/runtime.ts`, `public/worker/*.js` — 실행·Worker·Java `fetch` 경로
- `app/api/analyze/route.ts`, `app/api/java/execute/route.ts` — API 계약 (`/api/explain`은 아래 **삭제 예정** 정책에 따라 문서 검증 대상에서 제외)
- `src/hooks/useProvaExecution.ts`, `src/store/useProvaStore.ts` — 클라이언트 `fetch`·상태
- `src/lib/ai-providers.ts` — AI 프로바이더·폴백(문서에 "Gemini만" 등 단정이 있으면 코드와 일치하는지 확인)
- `env.example`, 배포 관련 env — 문서의 환경변수·URL 설명
- `CLAUDE.md` / `docs/features/*.md` 등 아키텍처 문서는 **상호 참조**하되, 최종 판단은 위 구현 파일에 맡긴다.

### 3. 문서 유형별 점검 항목

각 `.md`를 읽으며 아래를 **체크리스트**로 처리한다.

- **API 경로·메서드**: 실제 `app/api/**/route.ts` 존재 여부, 요청/응답 본문 필드명이 타입과 일치하는지.
- **클라이언트 네트워크**: "한 번의 디버깅에 `fetch` 몇 번" 등은 `useProvaExecution`·`runtime` 기준으로 검증.
- **Web Worker / postMessage / Java 원격 실행**: 역할 혼동(HTTP vs IPC) 문구 정정.
- **`/api/explain`**: 위 **「`/api/explain` 처리 원칙(삭제 예정)」** 절에 따라 문서에서 제거·비기능으로 정리한다. "코드에 파일이 있다"는 이유로 기능으로 서술하지 않는다.
- **AI 프로바이더**: `callWithFallback`·`AI_PROVIDER` 등과 모순되는 "단일 프로바이더만" 같은 단정 수정.
- **다이어그램(Mermaid 등)**: 아래 **§4 다이어그램 정합성**을 따른다.
- **폴더 구조·파일 경로**: 문서에 적힌 경로가 실제와 같은지 확인.

### 4. 다이어그램 정합성 (특히 신경 쓸 것)

문장 설명보다 **그림이 잘못되면 오해가 크게 퍼진다.** Mermaid 코드 블록 · 플로우차트 · 시퀀스 다이어그램 · ASCII 도식 등 **문서 내 모든 시각 자료**를 코드와 대조한다.

- **요청·응답 방향**: 누가 누구에게 `fetch`하는지(브라우저 ↔ Next ↔ 원격 JVM / Gemini). 클라이언트가 **LLM에 직접** 붙는 화살표, Next를 건너뛰는 화살표 등 **실제와 다른 화살표**를 제거·수정한다.
- **실행 경로**: Python·JavaScript(Web Worker·`postMessage`) vs Java(`fetch`·원격 JVM) 구분이 **한눈에 맞게**. "모든 언어가 동시에 도는 것처럼" 보이는 과도한 병렬 간선은 **오해 소지가 있으면** 정리한다(실제는 **언어 1개** 선택).
- **삭제 예정 API**: `/api/explain`은 다이어그램·범례에서 **빼거나**, 남기면 안 된다면 **비기능·삭제 예정**만 한 줄로 처리(위 explain 정책과 동일).
- **과장된 간선**: "거미줄" 수준의 교차선은 가능하면 **노드 통합·단계 축소**로 읽기 쉽게 하되, **사실은 바꾸지 않는다.**
- **수정 후**: Mermaid는 **렌더링 가능 여부**(문법)를 확인하고, 미리보기로 **화살표 방향**을 한 번 더 본다.

### 5. 수정 방식

- 사실 관계만 바로잡는다. **리팩터·기능 추가·불필요한 문장 장식**은 하지 않는다.
- 불확실하면 **코드 인용(파일 경로 + 확인한 동작)**을 문서에 짧게 남기거나, "현재 `src` 기준" 같은 한 줄 전제를 적는다.
- 프로젝트 규칙상 **사용자가 수동 관리하는 파일**(예: 루트 `README.md`, 기획서 등)이 있다면 **수정 전에 정책 확인**하거나 사용자에게 묻는다.

### 6. 산출물 — **가독성 좋은 보고서 필수**

작업을 끝낸 뒤, **에이전트 채팅(또는 이슈 본문)에 보고서를 반드시 제출**한다. 단순 나열이 아니라 **리뷰어가 스캔하기 쉬운 형태**로 쓴다.

권장 구조(제목·순서는 상황에 맞게 조정 가능):

1. **요약(3~8문장)**
   - 범위(`mocks/` 제외), 대략 **몇 개 `.md`를 열어보았고 몇 개를 수정**했는지, **다이어그램을 건드린 파일 수**
   - 코드는 **수정하지 않았음**을 한 줄로 명시

2. **변경 상세 표(핵심)**
   - 컬럼 예: `파일 경로` | `문제(문서가 틀렸던 점)` | `수정 내용(어떻게 고쳤는지)` | `근거(대조한 코드·경로)`
   - 수정이 없는 파일은 표에 넣지 않아도 되고, **변경이 있는 파일은 빠지지 않게** 모두 포함

3. **다이어그램만 손댄 경우**
   - 표에 같이 넣거나, **"다이어그램 변경" 소절**에서 파일별로 **무엇이 잘못된 화살표/노드였고 어떻게 바꿨는지**만 따로 요약

4. **미수정·보류**
   - 손대지 않은 이유(수동 관리 파일, 권한, 시간 등), **문서에 남긴 가정**

5. **코드에서 발견했으나 문서 작업으로 고치지 않은 것**
   - **별도 이슈 권장** 항목만 bullet(코드 수정은 하지 않았다는 점과 연결)

6. **검증 메모(선택)**
   - `grep`/파일 열람으로 확인한 근거를 한 줄씩만

- **변경 요약만 1~3문장으로 끝내지 말 것.** 위 수준의 **보고서가 산출물의 본체**다.
- **미해결·가정**: 코드가 애매하거나 환경별로 다른 부분은 문서에 남긴 내용과 함께 **보고서 §4(미수정·보류)** 에 적는다.

### 7. 검증(가능하면)

- 문서에서 언급한 API·타입명이 **grep으로 실제 존재**하는지 확인.
- **다이어그램**: Mermaid(또는 사용한 문법) **파싱 오류 없음**, 화살표·주석이 **위 다이어그램 점검 항목**과 모순되지 않음.

### 8. 커밋

수정된 `.md` 파일들을 `/commit` 커맨드를 사용하여 커밋한다.

## 금지·주의

- `mocks/` 내부는 **열지도 수정하지도 않는다.**
- **`/api/explain` 관련 문서를 새로 늘리거나 "구현됨"처럼 적지 않는다.** 삭제 예정·비노출 기능으로 맞춘다.
- **코드는 한 줄도 수정하지 않는다.** 이번 작업은 **문서 리팩터링**이며, 방향은 항상 **코드·실제 동작 → 문서**다. 문서가 틀렸으면 **`.md` 등 문서만 고친다**; 문서를 맞추려고 **실행 경로·API·타입·`src`/`app`/설정 스크립트 등 저장소의 비문서 소스를 손대는 행위는 전부 금지**다(리팩터, 버그 수정, 포맷, 주석 추가, 미사용 코드 삭제, 라우트·워커 수정 포함).
- 코드에 버그·불일치를 발견해도 **이 작업 범위에서 고치지 않는다.** 목록으로만 적고 **별도 이슈/후속 작업**으로 넘긴다.
- 추측으로 "아마 이렇게" 채우지 말고, **파일을 열어 확인**한다.

---

## 완료 정의

- `mocks/` 제외 전 `.md`에 대해 위 체크리스트를 적용했고, **코드와 모순되는 단정이 제거·수정**되었다.
- **다이어그램이 포함된 파일**은 **시각적 흐름도 코드와 맞게** 정리되었고, Mermaid 등 **문법이 깨지지 않았다.**
- **§6 산출물**에 해당하는 **보고서**를 제출했다(파일별 무엇을 어떻게 고쳤는지·근거가 **가독성 있게** 드러난다).
- 수정된 문서가 **커밋**되었다.
