# /docs-audit — Markdown 문서 전수 정합성 점검 + 수정

## 목적

`mocks/` 이하를 제외한 모든 Markdown 문서를 **현재 코드베이스 기준으로** 검토하고, 불일치/과장/구식 설명을 수정한다.
기존 Markdown의 주장은 **검증 전까지 신뢰하지 않는다.** 반드시 소스 코드/설정/타입 정의와 대조한다.
코드는 한 줄도 수정하지 않으며, 방향은 항상 **코드/실제 동작 -> 문서**다.

## 입력

- `$ARGUMENTS` — (선택) 특정 `.md` 파일 경로 또는 디렉토리. 미지정 시 저장소 전체(`mocks/` 제외) 스캔.

## `/api/explain` 처리 원칙

`POST /api/explain`은 **삭제 예정**이다. 저장소에 라우트 파일이 남아 있어도 문서 관점에서는 **없는 기능**으로 취급한다.

- 제품 기능/아키텍처/다이어그램/클라이언트 흐름에서 explain 관련 내용을 **삭제/축약**한다.
- "향후 연동", "라우트만 존재" 등 코드 존재를 전제로 한 설명도 지양하고, 분석은 `/api/analyze` 중심만 남긴다.
- 히스토리/내부 메모에 한 줄 남길 때만 "삭제 예정이었던 엔드포인트" 수준으로 적을 수 있다.

## 실행 절차

### 1. 인벤토리

- `mocks/` 밖의 모든 `.md` 파일 목록을 만든다(경로 + 대략적 역할 한 줄씩).

### 2. 코드 "진실의 원천" 확인

문서 주장과 대조할 때 아래를 우선 읽는다.

| 우선순위 | 파일 | 역할 |
|---|---|---|
| 1 | `src/types/prova.ts` | 파이프라인 타입/스키마 |
| 2 | `src/features/execution/runtime.ts`, `public/worker/*.js` | 실행/Worker/Java fetch 경로 |
| 3 | `app/api/analyze/route.ts`, `app/api/java/execute/route.ts` | API 계약 |
| 4 | `src/hooks/useProvaExecution.ts`, `src/store/useProvaStore.ts` | 클라이언트 fetch/상태 |
| 5 | `src/lib/ai-providers.ts` | AI 프로바이더/폴백 |
| 6 | `env.example`, 배포 관련 env | 환경변수/URL |

`/api/explain`은 삭제 예정이므로 문서 검증 대상에서 제외한다.

### 3. 문서별 점검 (체크리스트)

각 `.md`를 읽으며 아래를 적용한다.

- **API 경로/메서드**: 실제 `app/api/**/route.ts` 존재 여부, 요청/응답 필드명이 타입과 일치하는지.
- **클라이언트 네트워크**: fetch 횟수, 호출 순서 등을 `useProvaExecution`/`runtime` 기준으로 검증.
- **Web Worker / postMessage / Java 원격 실행**: 역할 혼동(HTTP vs IPC) 문구 정정.
- **`/api/explain`**: 위 삭제 예정 원칙에 따라 문서에서 제거/비기능으로 정리.
- **AI 프로바이더**: `callWithFallback`/`AI_PROVIDER` 등과 모순되는 "단일 프로바이더만" 같은 단정 수정.
- **폴더 구조/파일 경로**: 문서에 적힌 경로가 실제와 같은지 확인.

### 4. 다이어그램 정합성 (특히 엄격)

문장보다 **그림이 잘못되면 오해가 크게 퍼진다.** Mermaid/ASCII 등 모든 시각 자료를 코드와 대조한다.

- **요청/응답 방향**: 누가 누구에게 fetch하는지. 실제와 다른 화살표 제거/수정.
- **실행 경로**: Python/JS(Worker, postMessage) vs Java(fetch, 원격 JVM) 구분이 정확한지. 실제는 언어 1개 선택.
- **삭제 예정 API**: `/api/explain`은 다이어그램에서 제거.
- **과장된 간선**: 거미줄 수준의 교차선은 노드 통합/단계 축소로 정리하되 사실은 바꾸지 않는다.
- **수정 후**: Mermaid 문법 오류가 없는지 확인.

### 5. 수정 방식

- 사실 관계만 바로잡는다. 리팩터/기능 추가/문장 장식은 하지 않는다.
- 불확실하면 코드 인용(파일 경로 + 확인한 동작)을 문서에 짧게 남긴다.
- `README.md` 등 수동 관리 파일은 수정하지 않는다.

### 6. 보고서 출력

작업 완료 후 아래 구조의 보고서를 출력한다.

1. **요약** (3~8문장): 범위, 스캔/수정한 `.md` 수, 다이어그램 수정 파일 수, 코드 미수정 확인.
2. **변경 상세 표**: `파일 경로` | `문제` | `수정 내용` | `근거(대조한 코드/경로)`
3. **다이어그램 변경**: 잘못된 화살표/노드와 수정 내용 요약.
4. **미수정/보류**: 손대지 않은 이유, 문서에 남긴 가정.
5. **코드 이슈 발견**: 문서 작업으로 고치지 않은 코드 이슈(별도 이슈 권장).
6. **검증 메모**: grep/파일 열람으로 확인한 근거.

### 7. 커밋

변경된 문서를 아래 형식으로 커밋한다.

```
docs: Markdown 문서 코드 정합성 수정

- {변경 항목 1}
- {변경 항목 2}
- {변경 항목 3}
```

## 금지/주의

- `mocks/` 내부는 열지도 수정하지도 않는다.
- `/api/explain` 관련 문서를 새로 늘리거나 "구현됨"처럼 적지 않는다.
- **코드는 한 줄도 수정하지 않는다.** `.md` 등 문서만 고친다.
- 코드에 버그/불일치를 발견해도 이 작업 범위에서 고치지 않는다. 보고서에 목록으로만 적는다.
- 추측으로 채우지 말고, 파일을 열어 확인한다.
- `README.md`는 수동 관리 대상이므로 수정하지 않는다.
